# Claude.md — AI Civilization Simulation

## What We Are Building

A living, autonomous civilization simulation rendered in Three.js where every character and the world itself are AI agents. Two (expandable to n agents) agents are placed in a low-poly 3D world and left to autonomously build a civilization — foraging, constructing, specializing, coordinating, and surviving — while the user monitors from the outside. There is no game loop driven by the user. The agents drive everything. They build a society, with its own rules, norms, and culture of what they consider to be the perfect society.

---

## Core Philosophy

- Agents are autonomous. They decide when to act, what to do, who to talk to, and when to rest.
- The world is also an agent. It narrates, mutates state, and injects environmental pressure.
- The user is a monitor and occasional intervenor, not a player.
- All persistence is flat files on disk. No database.
- The LLM decides *what* to do. The server decides *what that means for the world*. These are never mixed.

---

## Architecture Overview

### Stack
- **Backend**: Node.js — single process, single world instance
- **Frontend**: Three.js — browser client, pure renderer, no logic
- **Transport**: WebSocket — server pushes state diffs to frontend
- **LLM**: Anthropic SDK (Claude) and/or OpenAI SDK (GPT-4o) — swappable per agent
- **Persistence**: Flat markdown and JSON files on disk

### Process Layout
One Node.js process runs everything:
- All Character instances
- The World Agent instance
- The Event Bus (EventEmitter, upgradeable to Redis)
- The WebSocket server
- The Tick/Action scheduler

No containers needed for local dev. Deployable as-is to a single VPS (Railway, Fly.io, DigitalOcean) when ready.

## The Character Class

Each agent is an instance of `Character`, loaded from its folder at boot:

```js
const alice = new Character("alice")  // loads all md files from agents/alice/
const bob   = new Character("bob")
```

The class:
- Composes system prompt by interpolating: `claude.md + personality.md + beliefs.md + last 10 lines of memory.md + current goals.md + perception packet from World Agent`
- Maintains its own rolling message history (last ~20 messages, sliding window)
- Exposes `wake(event)` — called when a relevant event arrives on the bus
- Makes LLM API call, receives a response that may include tool calls
- Publishes resulting actions/messages back onto the Event Bus
- Model is configurable per agent (`claude-sonnet-4-5`, `grok-4-1-fast-reasoning`, `gpt-4o`, etc.)

The World Agent is the same pattern but reads from `world/` and has authority over `state.json`.

## Agent Harness — System Prompt Stack

Injected fresh on every wake, in this order:

```
[WORLD RULES & ACTION VOCABULARY]     ← claude.md
[YOUR PERSONALITY & BACKSTORY]        ← personality.md
[YOUR LONG-TERM BELIEFS]              ← beliefs.md
[YOUR CURRENT GOALS]                  ← goals.md
[YOUR RECENT MEMORIES]                ← last 10 lines of memory.md
[WHERE YOU ARE & WHAT YOU PERCEIVE]   ← perception packet (generated by World Agent)
[YOUR CURRENT STATE]                  ← activity / reachability / disposition
```

## Agent State System

Every agent carries three orthogonal state axes:

| Axis | Values |
|---|---|
| **activity** | `active` / `non-active` |
| **reachability** | `listening` / `non-listening` |
| **disposition** | `working` / `socializing` / `exploring` / `resting` / `alert` |

### Valid Combinations
- `active / listening / working` — focused on tasks, minimal conversation
- `active / listening / socializing` — present with others, conversation-heavy, low action bias
- `active / listening / exploring` — roaming, moderate action, low conversation
- `active / non-listening / working` — solo deep work, won't respond to agent pings
- `non-active / listening / resting` — dormant but world/priority events can wake them
- `non-active / non-listening` — **death only**

State is tracked server-side and injected into the system prompt. The agent changes state via tool call. The World Agent is notified of all state transitions.

### Disposition Behavioral Contracts (in claude.md)
Each disposition carries an explicit behavioral instruction so the agent self-regulates:
- `working` → prioritize action tool calls, keep conversation minimal
- `socializing` → conversation is the activity, actions are low priority
- `resting` → no self-initiated actions, brief responses only if woken
- `alert` → ignore all non-urgent inputs, react to threat immediately

## Event-Driven Wake System

There is no fixed tick interval. Agents wake up when something happens that is relevant to them. Events flow through the Event Bus:

### Event Types
- `WORLD_EVENT` — environmental update from World Agent (weather, time, passive mutation)
- `AGENT_ACTION` — an agent completed or started an action
- `AGENT_MESSAGE` — direct p2p message from one agent to another
- `POSITION_UPDATE` — agent moved zones
- `ACTION_COMPLETE` — multi-tick action finished
- `USER_INJECT` — user intervention from the monitoring panel
- `PRIORITY_EVENT` — bypasses reachability/activity gates (threat, emergency)

### Routing Rules
- Events are routed to agents based on their current state
- `non-active/listening` agents receive `WORLD_EVENT` and `PRIORITY_EVENT` only
- `active/non-listening` agents receive `ACTION_COMPLETE` and `PRIORITY_EVENT` only
- `active/listening` agents receive all event types
- P2P messages to non-listening agents are queued and optionally narrated by World Agent

## Memory System

### Three Layers

**Layer 1 — Working memory (RAM)**
Rolling message history, last ~20 exchanges. Ephemeral.

**Layer 2 — Episodic memory (memory.md)**
Append-only log the agent writes to itself via `write_memory` tool call — only when it judges something worth remembering, guided by claude.md instructions. Not every tick. Only meaningful events.

```
[event 42] Walked to the river. Alice was already there.
[event 47] Alice warned me about the northern forest. Said something moved there.
[event 63] Completed the first wall section. Took longer than expected.
```

**Layer 3 — Belief consolidation (beliefs.md)**
Periodically regenerated (every N events or on session end) by summarizing `memory.md` into stable, concise beliefs. Always injected into system prompt. Short — 10 to 20 bullet points.

### Memory Write Gate
Before writing, the agent is shown its last 5 entries and instructed: if this is semantically identical to recent entries, do not write — take an action instead. Prevents memory log spam and loop reinforcement.

## Spatial System

### Zone Graph
The world is a coordinate grid. Zones are defined in `state.json`:

```json
{
  "river_bank": {
    "description": "Mossy bank beside slow water",
    "adjacent": ["forest_edge", "stone_bridge"],
    "objects": ["fallen_log", "fishing_rod"],
    "structures": [],
    "agents": ["alice"]
  }
}
```

### Perception Packet
On each wake the World Agent compiles a natural language spatial context for the agent:
- Where you are, what is here, who is here, What you can see, what you can hear, what you can smell
- What is in adjacent zones (and who)
- What you can hear from further zones
- Recent world events visible from this position

### Spatial Query Tool
Agents can call `query_space({ from, direction })` before moving. World Agent responds in prose: what they would find, who is there, what zones open up from there. Pure graph traversal dressed in language.

### Three.js Mapping
Frontend maintains a `ZONE_POSITIONS` lookup. Zone names → 3D coordinates. When `state.json` changes, the server broadcasts diffs. Three.js lerps characters and renders/removes structure meshes. Semantic and visual layers are fully decoupled.

## Action System

### Action Registry (actions.json)
Every possible action in the world is defined here with: description, requirements (tool, object, proximity, tech), duration in ticks, what it produces/consumes/destroys, and what fields of `state.json` it mutates.

### How Agents Act
1. Agent calls `act({ action: "chop_wood", target: "oak_1" })`
2. Server validates against `actions.json` (requirements met?)
3. If multi-tick: queued in pending actions, agent mode set to `active/non-listening`
4. On completion: deterministic mutation function runs on `state.json`
5. World Agent narrates the result, broadcasts to listening agents
6. Three.js receives mutation diff and updates scene

### Mutation Functions
Pure deterministic JS functions on the server. The LLM never decides how the world changes — only what action to attempt. Mutations are never hallucinated.

### Production Chains
Actions chain — outputs of one action are inputs to another:
`tree → logs → planks → walls → house → settlement`
`wheat → grain → flour → bread`
Built structures unlock new actions and accelerate existing ones (sawmill speeds wood processing, farm produces food passively).

## World Agent

Same harness pattern as Character but with world-level authority.

Responsibilities:
- Generates perception packets for all agents on request
- Validates and applies action mutations
- Narrates world events in natural language
- Manages `civilization.md` — the shared history ledger
- Runs passive world processes (crop growth, decay, season changes, weather)
- Injects environmental forcing events when agents are looping (no action events in N events → fire a world event)
- Receives all state transition notifications

---

## Civilization Layer

### Shared State
- `world/civilization.md` — growing ledger of what has been built, discovered, lost
- `world/tech.json` — unlocked technologies that gate new actions
- Agent `goals.md` files include both personal goals and current civilization goals

### Production & Compounding
Built structures create passive effects: farms produce food per passive tick, sawmills accelerate wood actions, walls change zone traversability. The civilization compounds over time.

### Specialization
Agent `skills` are tracked implicitly through memory — what they've done most. The perception packet surfaces this: *"Bob has built 4 structures. You have done most of the foraging."* Agents self-specialize based on identity.

### Scarcity & Stakes
Passive world processes create pressure: winter reduces food availability, storms damage unprotected structures, untended crops fail. Agents have failure conditions that make urgency real.

---

## Anti-Loop Mechanisms

1. **Goals.md** — agents always have an active goal list; no goal = no north star = drift
2. **Disposition contracts** — `working` mode explicitly deprioritizes conversation
3. **World Agent forcing events** — if only `AGENT_MESSAGE` events for N consecutive events with no `ACTION` events, World Agent fires an environmental interrupt
4. **Memory write loop check** — agent sees last 5 entries before writing; if looping, it's instructed to act instead
5. **Conversation structure** — claude.md instructs: every exchange should terminate in an action or a question requiring a factual answer; no open-ended commentary

## User Intervention

The user connects via a monitoring panel (separate from Three.js or overlaid). They can:
- Inject a `USER_INJECT` event (delivered as a `PRIORITY_EVENT` to all listening agents)
- View live agent state, disposition, current goals, recent memory
- Pause/resume the world
- Manually trigger a World Agent environmental event

---

## Key Design Contracts

- **LLM decides what. Server decides what that means.** Mutations are never left to the model.
- **World Agent is the only spatial authority.** Agents perceive space through it, never directly.
- **All state is files.** Restart = resume. No database needed.
- **Events are the only inter-agent channel.** No direct class method calls between agents.
- **State transitions are always tool calls through the bus.** The world always knows.
